- **Module 15 — Marks & Result Processing** (2026-07-22): the back half of the exam cycle. Mark entry runs a **four-eyes lifecycle** (`DRAFT→SUBMITTED→VERIFIED→LOCKED`, each step its own permission because they are meant to be different people); a paper moves as a unit, a save is **all-or-nothing** with every bad cell returned at once, and a LOCKED mark leaves the flow entirely — it changes only through the **correction flow**, which demands a reason, writes an unerasable `mark_corrections` row *before* the mark moves, and re-processes that candidate. The bound a DB CHECK cannot express — a mark against its component's allocation, one join away on `exam_subjects` — lives in `mark-entry.engine.ts`, mirrored client-side so cells turn red before a request is sent. **Five dependency-free engines**, golden-tested against hand-computed NCTB values: the frozen scale, subject grading (a **missed component threshold forces F even on an 88/100**), NCTB GPA (mean of *grade points*; the 4th subject contributes only points **above** the base and never enters the divisor; one compulsory F is GPA **0.00**, not an average; an optional F never fails anybody), competition ranking (1, 2, 2, 4, among PASSED only) and the weighted merge. **Grace marks** finally consume the M04 placeholder setting: capped per subject *and* in how many subjects, recorded separately from the entered mark, printed on the report card. **Processing runs** are idempotent upserts with durable progress in `result_runs` (so the status endpoint survives a Redis restart), an issue list the UI renders, and merit as a **second pass** because positions are relative. **Two design corrections to M14**: the grade scale is now frozen at **first PROCESSING, not at PUBLISH** — results were otherwise graded through the live table, and a band edit before publication would freeze a scale no result was computed against; and **publication visibility is the active `result_publications` row, not `exams.status`**, because the status machine deliberately cannot rewind past PUBLISHED. Versioned publications (partial unique index keeps exactly one active), report cards / tabulation XLSX+PDF / transcripts, analytics (GPA histogram, subject difficulty, year-over-year by exam **type**), combined results with the weight set frozen per row, and a `@Public` website result search where a withheld result and a non-existent student return the *same* 404. **13 permission codes, 9 new `exam.*` settings.** Frontend: three new tabs on `/admin/exams/[id]` (keyboard-first mark grid where Enter/↑/↓ move down the *column*, results + processing + publish dialog, analytics) and `/admin/results/combined`. **Closed six cross-module debts**: `EXAM_RESULT_GATE` is live, the M14 delete guards on `exam_classes`/`exam_subjects` bite, M06 subject-removal is blocked once marks exist, the M11 promotion rollback guard extends to marks, and M09 `performance-history` returns real data. 808 backend unit (219 new) + **55-case `result.e2e-spec.ts`** taking e2e to 280 tests / 15 suites / 195 frontend (24 new) tests green; migration verified on an empty Postgres 16 **and** Neon with zero drift.
